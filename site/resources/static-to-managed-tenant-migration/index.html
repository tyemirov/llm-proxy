Static-to-managed tenant migration in management mode

Migrate legacy config tenants once, then make database state authoritative for managed runtime access.

Operators moving from static tenant YAML to self-service management mode.

Open LLM Proxy Browse all resources

The problem

Static tenants and managed tenants can conflict if old YAML keeps influencing runtime after users start managing secrets and provider keys in the database.

How LLM Proxy helps

On first management-mode startup, LLM Proxy migrates legacy config tenants and nonblank configured provider keys into the database, records a migration marker, and then treats DB state as authoritative for runtime authentication and routing.

How it works

  1. Enable management mode with database settings.
  2. Start the service so the one-time migration can run.
  3. Confirm managed tenant state exists in the database.
  4. Remove migrated tenants and provider api_key values from config after confirmation.

Feature-to-benefit table

Feature Why it matters here Example
One-time migration marker After migration, config-file tenants and provider keys are obsolete for runtime auth/routing. The current contract is database-owned.
Provider key carryover Nonblank configured provider keys can seed managed provider records. Existing setup can move into management mode.
Forward-only cleanup Docs tell operators to remove migrated config state. No dual-read compatibility layer remains.

Use-case examples

Initial management rollout

An existing static tenant becomes a managed database tenant on first startup.

Provider key migration

A configured OpenAI key moves into encrypted managed storage.

Post-migration cleanup

Operators delete stale tenant secrets from config.yml after verifying DB state.

Objections and limitations

  • Migration is intended as a bounded move into the current schema.
  • The management database must be available and configured.
  • Server settings, provider URLs, model catalogs, and browser config remain config-owned.
