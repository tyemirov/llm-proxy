Legacy token ownership migration in management mode

Claim one prior static-config token for a verified user account without changing the token or losing usage history.

Operators retiring the final unowned management-mode token after moving to self-service accounts.

Open LLM Proxy Browse all resources

The problem

A token imported by an older release can still belong to a synthetic static-config user, so its real owner cannot see that token's usage after signing in.

How LLM Proxy helps

LLM Proxy rejects the unowned token, then atomically replaces any empty account created by an earlier sign-in and rekeys the legacy tenant, encrypted provider settings, and usage events when the configured owner email signs in through TAuth.

How it works

  1. Configure the exact legacy tenant id and deployment-owned target email.
  2. Deploy the current binary and drain older service instances before the owner claim.
  3. Sign in with the configured email and verify the same token plus historical usage.
  4. Remove the temporary migration config after production verification.

Feature-to-benefit table

Feature Why it matters here Example
Verified owner claim Only a live TAuth session whose normalized email matches the configured owner can claim the tenant. The migration does not trust stored email alone.
Token and usage continuity The token digest, tenant defaults, creation time, and every usage event are preserved. Existing clients keep their token while the dashboard gains its history.
Provider key re-encryption Provider ciphertext is decrypted under the synthetic user id and re-encrypted for the TAuth subject inside one GORM transaction. Ownership changes without copying invalid ciphertext.

Use-case examples

Unowned token before claim

The legacy token returns 403 instead of continuing as a global system credential.

First matching sign-in

The configured owner opens the management site and receives the existing tenant and usage history.

Post-migration cleanup

Operators remove the temporary owner mapping after verifying the unchanged token in production.

Objections and limitations

  • The source must be the one configured static-config tenant; an existing destination is replaceable only when it has no generated secret, provider settings, or usage.
  • A populated destination conflict returns 409 without merging or overwriting account state.
  • Old service instances must be drained before the owner signs in.
