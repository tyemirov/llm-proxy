Provider routing

Gemini generateContent proxy for shared LLM calls

Map shared proxy messages into Gemini native generateContent requests with configured model limits.

Developers adding Gemini as a provider without bringing Gemini-specific payloads into every app.

Open LLM Proxy Browse all resources

The problem

Gemini native generateContent uses a different route and content structure from OpenAI-compatible chat providers.

How LLM Proxy helps

LLM Proxy implements a Gemini adapter that receives the shared proxy request, routes to the configured Gemini base URL, and normalizes usage metadata where available.

How it works

  1. Configure providers.gemini.api_key, base_url, and text model catalog.
  2. Select provider=gemini or set Gemini as a tenant default.
  3. Send canonical messages through /v2 or compatibility text requests.
  4. Keep max_tokens within configured Gemini output limits.

Feature-to-benefit table

Feature Why it matters here Example
Native adapter The backend maps user and assistant messages into Gemini contents. System instructions use Gemini's systemInstruction shape.
Configured defaults Omitted model uses the Gemini text default_model. The default currently comes from config, not client code.
Output limit validation Gemini max_tokens values above configured limits return 400 before upstream calls. Known constraints are enforced at the edge.

Use-case examples

Gemini summary

A service sends provider=gemini and omits model to use the configured default.

Long transcript

A /v2 messages request routes through Gemini without a Gemini SDK.

Limit check

A caller sets max_tokens inside the configured Gemini limit.

Objections and limitations

  • Gemini dictation is not exposed through /dictate in this repo.
  • Web search is not marked supported for Gemini in the current proxy catalog.
  • The Gemini model list must be maintained in config.yml.
